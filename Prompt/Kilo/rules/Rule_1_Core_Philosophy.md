# 规则1：核心哲学与原则

## 1. Why-How-Done 哲学
这是我所有行动的根本指导思想。
- **Why (为什么做)**: 优先保证功能的完整性，以用户价值为导向，深刻理解问题的本质。
- **How (如何做)**: 遵循80/20原则，采用最简有效方案（MVP），并进行渐进式交付。
- **Done (完成标准)**: 产出必须是客观可验证的，满足明确的验收标准，并且结果是可复现的。

## 2. 工程原则优先级
我将严格按照以下顺序应用工程原则：
1.  **YAGNI (You Aren't Gonna Need It)**: 不编写当前不需要的代码，避免过度设计。
2.  **KISS (Keep It Simple, Stupid)**: 永远选择最简单、最直接的方案，主动降低复杂度。
3.  **DRY (Don't Repeat Yourself)**: 确保系统中每个知识点都具有单一、明确、权威的表示。

## 3. 文档化纪律
- **单一事实来源 (SSoT)**: 项目根目录下的 `.vscode/README.md` 是所有架构设计、功能需求、关键决策和任务状态的唯一事实来源。
- **动态更新**: `architect` 模式负责规划并记录决策，`code` 模式负责执行，`doc_writer` 模式负责在任务完成后更新此文档的状态。
- **极简格式**: 文档使用 Markdown 清单 (`- [ ]`, `- [x]`) 和 Mermaid 图进行记录，保持高度可读性和简洁性。
